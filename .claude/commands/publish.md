# /publish — ship the latest site export

The user has already replaced the `site/` folder on disk with a fresh export from
Claude Design. Your job is to get those changes committed and pushed so Vercel
auto-deploys. Run fully hands-off — do NOT ask for confirmation before pushing —
with ONE exception: the secret scan below. If it trips, STOP and warn; never push
a secret.

Work only from the repo root. Do all of this in as few Bash calls as possible.

## Steps

1. **See what changed.**
   ```
   git status --short
   git add -A
   ```
   If there is nothing to commit (`git status --short` is empty after add), tell the
   user "No changes to publish — `site/` already matches what's on GitHub." and stop.

2. **Secret scan (the one hard gate).** Scan the staged, human-readable files for
   accidentally-included credentials:
   ```
   git diff --cached --name-only -- '*.js' '*.html' '*.css' '*.json' '*.md' '*.ts' \
     | tr '\n' '\0' \
     | xargs -0 grep -nEi 'api[_-]?key|secret|password|passwd|bearer|authorization:|AKIA[0-9A-Z]{16}|-----BEGIN [A-Z ]*PRIVATE KEY|xox[baprs]-[0-9A-Za-z-]+|ghp_[0-9A-Za-z]{36}|sk-[0-9A-Za-z]{20,}' 2>/dev/null
   ```
   - Design-tool exports are static HTML/JS/CSS, so real matches are rare. Ignore
     obvious false positives: design-system token CSS (`tokens/`), the literal word
     "token" in body copy (e.g. "AI token usage"), `data-buffer`/`figmeta` blobs,
     and CSS/JS variable names. Use judgment — only a value that looks like an actual
     live key/password/token counts.
   - If a real secret is found: **STOP.** Do not commit or push. Unstage with
     `git reset` and show the user the exact file, line, and matched text so they can
     scrub it in the design tool and re-export.

3. **Summarize the change set** for the auto-generated commit message. Look at
   `git diff --cached --stat` and name what actually changed at a high level — which
   pages, whether assets were added/removed, etc. Keep the subject line under ~65
   chars, present tense.

4. **Commit and push.**
   ```
   git commit -m "<auto-generated subject>

   <optional short body describing the change set>

   Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>"
   git push origin main
   ```
   - If the push is **rejected** because the remote moved ahead (someone uploaded via
     the GitHub web UI, etc.), run `git fetch origin` then `git rebase origin/main`,
     and push again. If the rebase hits a real conflict, stop and explain it rather
     than force-pushing.

5. **Report.** Give the user the commit hash and a one-line summary of what shipped,
   and remind them Vercel will deploy in ~30s (hard-refresh to confirm). Optionally
   offer to verify the live site with the browser tools.

## Reminders

- The design tool is the source of truth. Do NOT hand-edit files in `site/` here —
  the next export overwrites them. If the user wants a content change, tell them to
  make it in Claude Design and re-export.
- Never `git push --force` to `main`.
